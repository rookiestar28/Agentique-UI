# Release Publication Policy

Status: public repository policy, release remains evidence-gated

Agentique UI is published as a public source repository. The Agentique web and catalog systems remain separate. Private planning records, operator notes, and unreleased evidence are not part of this repository.

## Repository Visibility

- Repository source may be public.
- Runtime secrets, signing keys, certificates, release credentials, and local smoke logs must never be committed.
- Public files must use product and engineering language only. They must not include private planning identifiers, local machine paths, or private evidence labels.

## Package Publication

The npm package is intentionally marked private. This prevents accidental registry publication while still allowing the source repository to be public.

Desktop distribution uses signed installer and updater artifacts, not npm package publication. A registry package release requires a separate public decision, package scope review, and release evidence.

## License Status

The repository uses the Apache-2.0 source license. This license applies to the public Agentique UI source repository and does not grant rights to separate Agentique web/catalog system contents, signing credentials, platform infrastructure, or unpublished release evidence.

## Versioning

Agentique UI uses semantic versioning for public release candidates and public releases.

| Channel | Example | Meaning |
|---|---|---|
| Local development | `0.1.0-dev` | Developer-only validation state |
| Release candidate | `0.1.0-rc.1` | Draft release review, not public install claim |
| Stable release | `0.1.0` | Published only after all release gates pass |

Release tags use `v` plus the version, for example `v0.1.0`. Release notes must describe only evidence-backed behavior.

## Public Naming Rules

Public branch names, tags, release names, workflow labels, commit messages, and docs must avoid:

- private planning identifiers,
- private evidence locations,
- local absolute paths,
- credentials or secret-like placeholders,
- installer, updater, or runtime claims before matching artifacts and smoke evidence exist.

## Release Authority

A draft release may be generated by automation. A stable public release requires maintainer review of installer artifacts, updater metadata, checksums, provenance, smoke evidence, public documentation, and rollback readiness.
